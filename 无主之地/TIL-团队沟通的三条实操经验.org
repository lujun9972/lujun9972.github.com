#+DESCRIPTION: 团队里"说一遍"远远不够。三条经验：重复到你自己都烦，用不同渠道传递不同粒度的细节，问"你怎么想的"而非"清楚了吗"。

* 是什么

你是不是遇到过这么一个情况：在团队里宣布了一个新流程或新方案，过了一周发现没人按新方式做事。针对这种情况，Roman Dzvinskyi 在 [[https://leadershipintech.com/links/22315/11c42bc5-a9ca-40e4-aae5-3ff137b7d6a0/email][High-Impact Engineering 通讯]] 中给了三条实操建议：重复到你自己都烦，用不同渠道传递不同粒度的细节，问"对方的想法"而非"清楚了吗"。

* 为什么

管理者一个常见误解是以为在会上讲了一遍，就默认所有人都理解了。实际上每个人的理解程度都不一样。有人没听到，有人听到了但理解偏了，有人理解了字面意思但没领会隐含的意义，还有人在会上频频点头但扭头就把这件事给忘了。

原文认为信息传达有四个阶段：不知道、知悉、熟悉、采纳。不同的目标需要把人推到不同的阶段。比如，通知性信息（"下周二 10 点到办公室参加培训"）推到"知悉"就够了。但像新的工作模式这类复杂话题，需要团队成员不只是听懂，还要真的在日常工作中用起来，这光靠说一两遍远远不够。

* 怎么做

1. *重复到你自己都烦*（repeat until it hurts）。简单的通知可能重复三四次就够，但像组织愿景这样的信息可能需要重复几十次。重复有复利效应：先理解的人可以帮助后来者理解。
2. *用不同渠道传递不同粒度的细节*。比如在团队会议上介绍新的 code review 流程的思路和原因，会后发一条飞书/钉钉摘要方便回看，摘要里链接到包含技术细节和最佳实践的文档。同一话题，通过不同媒介，传递不同深度的内容。
3. *别问"清楚了吗"，问"你怎么想的"*。"清楚了吗"几乎必然得到"清楚了"的回答，因为没人想当那个没听懂的人。改成"你觉得这个 code review 流程改了之后会怎么跑？"，既给了对方安全表达困惑的空间，也能帮你发现哪些表述需要调整。
