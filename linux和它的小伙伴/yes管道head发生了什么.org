* 提出问题

让我们先看一个简单的命令：

#+BEGIN_SRC shell
yes | head
#+END_SRC

输出结果：

#+BEGIN_EXAMPLE
y
y
y
y
y
y
y
y
y
y
#+END_EXAMPLE

这里 =yes= 命令会不断输出 =y= （默认情况下），而 =head= 命令默认只取前 10 行。问题来了：为什么这个管道命令不会一直运行下去？

毕竟 =yes= 是个"不听话"的程序，它会无限地输出 =y= ，理论上应该一直写下去才对。但实际上，命令执行完 10 行后就正常退出了。这是为什么？

* Shell 做了什么

要理解这个问题，首先需要看看 Shell 在遇到管道符号 =|= 时做了什么。

当 Shell 执行 =yes | head= 时，它会经历以下步骤：

1. 调用 =pipe()= 系统调用创建一个管道，得到两个文件描述符：=fd[0]= （读端）和 =fd[1]= （写端）
2. 调用 =fork()= 两次，创建两个子进程
3. 在 =yes= 进程中：
   - 关闭读端 =fd[0]=
   - 使用 =dup2(fd[1], 1)= 将标准输出（stdout）重定向到管道写端
   - 关闭原来的 =fd[1]=
   - 执行 =yes= 程序
4. 在 =head= 进程中：
   - 关闭写端 =fd[1]=
   - 使用 =dup2(fd[0], 0)= 将标准输入（stdin）重定向到管道读端
   - 关闭原来的 =fd[0]=
   - 执行 =head= 程序

可以用一个 ASCII 图示来表示这种连接关系：

#+BEGIN_EXAMPLE
  Shell (bash)
    │
    ├─ pipe() ──▶ [fd[0] 读端, fd[1] 写端]
    │
    ├─ fork() ──▶ yes 进程
    │               关闭 fd[0]
    │               dup2(fd[1], 1)  ← stdout → 管道写端
    │
    └─ fork() ──▶ head 进程
                    关闭 fd[1]
                    dup2(fd[0], 0)  ← 管道读端 → stdin

  最终效果：

  yes ──stdout──▶ [ 内核管道缓冲区 ] ──stdin──▶ head
#+END_EXAMPLE

让我们用 =strace= 来观察一下实际的系统调用序列：

#+BEGIN_SRC shell
strace -f -e trace=pipe2,clone,dup2,close,execve sh -c 'yes | head' 2>&1 | head -20
#+END_SRC

输出（经过简化整理）：

#+BEGIN_EXAMPLE
pipe2([3, 4], 0)                  = 0        ← 创建管道，fd[0]=3(读), fd[1]=4(写)
clone(...)                        = 45210    ← fork 出 yes 进程
close(4)                          = 0        ← bash 关闭写端（自己不写）
clone(...)                        = 45211    ← fork 出 head 进程
close(3)                          = 0        ← bash 关闭读端（自己不读）

45210 close(3)                    = 0        ← yes 关闭读端
45210 dup2(4, 1)                  = 1        ← yes: fd[1] → stdout
45210 close(4)                    = 0        ← yes 关闭原来的 fd[1]
45210 execve("/usr/bin/yes" ...)  = 0        ← yes 开始执行

45211 dup2(3, 0)                  = 0        ← head: fd[0] → stdin
45211 close(3)                    = 0        ← head 关闭原来的 fd[0]
45211 execve("/usr/bin/head" ...) = 0        ← head 开始执行
#+END_EXAMPLE

从 =strace= 输出可以看到，Shell 确实按照我们描述的步骤建立了管道连接。现在 =yes= 的 stdout 连接到管道写端，=head= 的 stdin 连接到管道读端。

但这还没有解释为什么 =yes= 会停止输出。让我们继续往下看。

* 管道是什么

在深入理解进程协作之前，我们需要先了解管道的本质。

管道并不是什么神奇的东西，它本质上就是内核维护的一个 *固定大小的缓冲区* 。在 Linux 中，这个缓冲区默认大小是 64KB（可以通过 =fcntl(fd, F_GETPIPE_SZ)= 系统调用查询）。

管道有几个重要的特性：

1. *先进先出（FIFO）* ：先写入的数据会被先读出，就像排队买票一样
2. *数据被读走后即从缓冲区中移除* ，腾出空间给新数据
3. *单向流动* ：数据只能从写端流向读端，不能反向
4. *匿名管道* ：没有名字，只能用于有亲缘关系的进程之间（如父子进程）

可以把管道想象成一根水管：

#+BEGIN_EXAMPLE
  yes 进程                     内核管道                     head 进程
    │                                                          │
    │  写入 y ────▶  [ y | y | y | y | ... | y ]  ────▶  读取 y
    │                                                          │
                  ▲                 │
                  │                 ▼
               写端入口          读端出口
             (只能进)           (只能出)
#+END_EXAMPLE

这个水管（缓冲区）有固定的容量。当水满了（缓冲区满了），就不能再往里倒水（写入数据）；当水空了（缓冲区空了），就取不出水（读取数据）。

* 运行中的协作

现在我们终于可以回答最初的问题了：为什么 =yes= 会停止输出？

关键在于 *blocking I/O* 的协调机制。（关于 blocking I/O 的详细原理，我已经在另一篇文章中详细讨论过，这里只概述结论）。

当进程通过管道协作时，内核会自动协调它们的速率：

1. *当缓冲区满了* ：=yes= 的 =write()= 系统调用会阻塞，=yes= 进程进入睡眠状态，等待管道腾出空间
2. *当缓冲区空了* ：=head= 的 =read()= 系统调用会阻塞，=head= 进程进入睡眠状态，等待新的数据到来

这种机制就像一个自动调节阀门，让快慢不同的进程能够和谐地协同工作。

不过，在我们这个具体的场景中，情况有些特殊。=yes= 的输出速度 *远快于* =head= 的消耗速度（=head= 只需要 10 行就满足了）。所以实际情况是：

1. =yes= 飞速地往管道里写 =y=
2. =head= 慢悠悠地从管道里读取（只读了 10 行）
3. =head= 读够 10 行后就退出了，*关闭了管道的读端*
4. 当 =yes= 下次尝试写入时，内核发现 *读端已经关闭* ，于是向 =yes= 发送 =SIGPIPE= 信号（或让 =write()= 返回 =EPIPE= 错误）
5. =yes= 收到信号后终止运行

所以，真正让 =yes= 停下来的不是缓冲区满了，而是 *读端的关闭* 。

这就是管道命令能够正常退出的完整机制。两个进程通过内核提供的管道和 blocking I/O 机制，实现了完美的协作。

* head 完成后发生了什么

上一节我们提到了一个关键结论：=yes= 进程之所以终止，是因为 =head= 关闭了管道的读端。但这只是概述，让我们深入看看这个过程究竟是如何一步步发生的。

这个终止链条实际上是一个精妙的连锁反应，可以分解为以下四个步骤：

**第一步：head 读够了 10 行**

=head= 命令默认只读取前 10 行数据。一旦读够了，它的任务就完成了。

#+BEGIN_EXAMPLE
  yes ──stdout──▶ [ 管道缓冲区 ] ──stdin──▶ head ✓ (完成)
#+END_EXAMPLE

**第二步：head 进程退出，内核关闭管道读端**

当 =head= 进程调用 =exit()= 或 =exit_group()= 系统调用退出时，内核会自动关闭该进程打开的所有文件描述符，包括管道的读端（stdin）。

#+BEGIN_EXAMPLE
  yes ──stdout──▶ [ 管道缓冲区 ] ──✕── (读端已关闭)
#+END_EXAMPLE

**第三步：yes 继续写入，内核发现没有读端**

此时 =yes= 进程还在不知情地继续调用 =write()= 往管道里写数据。但内核发现管道的读端已经没有进程在持有了（所有引用都关闭了）。

内核向 =yes= 进程发送 =SIGPIPE= 信号（信号编号 13），并且让 =write()= 系统调用返回 =EPIPE= 错误。

#+BEGIN_EXAMPLE
  yes ──write()──▶ ???  ← 内核发送 SIGPIPE (信号13)
#+END_EXAMPLE

**第四步：SIGPIPE 默认行为：终止 yes 进程**

=SIGPIPE= 信号的默认处理方式是 *终止进程* 。=yes= 没有特意设置信号处理函数，所以它收到了 =SIGPIPE= 后就直接退出了。

#+BEGIN_EXAMPLE
  yes ✕ (被 SIGPIPE 杀死)
#+END_EXAMPLE

让我们用 =strace= 来观察这个过程，看看实际的系统调用序列：

#+BEGIN_SRC shell
strace -f -e trace=write,close,exit_group,sigaction sh -c 'yes | head' 2>&1 | tail -20
#+END_SRC

输出（经过简化整理，进程 ID 45210 是 =yes= ，45211 是 =head= ）：

#+BEGIN_EXAMPLE
45210 write(1, "y\ny\n...", 8192)       = 8192     ← yes 正常写入
45210 write(1, "y\ny\n...", 8192)       = 8192     ← yes 正常写入
45210 write(1, "y\ny\n...", 8192)       = 8192     ← yes 正常写入
45211 close(0)                          = 0        ← head 关闭 stdin（管道读端）
45210 write(1, "y\ny\n...", 8192)       = -1 EPIPE ← yes 写入失败：EPIPE（断开的管道）
45210 --- SIGPIPE {si_signo=SIGPIPE, si_code=SI_USER, si_pid=45210, si_uid=1000} ---
45210 +++ killed by SIGPIPE +++
45211 write(1, "y\ny\n...", 20)          = 20       ← head 输出结果到终端
45211 exit_group(0)                     = ?        ← head 正常退出
#+END_EXAMPLE

这个 =strace= 输出完美地展示了我们描述的终止链条：

1. =yes= 进程（45210）连续调用 =write()= 正常写入数据
2. =head= 进程（45211）调用 =close(0)= 关闭了 stdin（管道读端）
3. =yes= 下次调用 =write()= 时返回了 =-1 EPIPE= 错误
4. =yes= 收到 =SIGPIPE= 信号，被杀死（=killed by SIGPIPE= ）
5. =head= 输出结果并正常退出（=exit_group(0)= ）

这里有一个细节值得注意：=close(0)= 发生在 =write()= 返回 =EPIPE= *之前* 。这说明 =head= 关闭读端是主动行为，而 =yes= 的写入失败是被动的后果。

如果你想让 =yes= 不被 =SIGPIPE= 杀死，可以忽略这个信号。例如：

#+BEGIN_SRC shell
trap '' PIPE; yes | head
#+END_SRC

这样 =yes= 的 =write()= 会返回 =EPIPE= 错误，但 =yes= 不会退出。不过，大多数程序都不会特意处理 =SIGPIPE= ，所以默认行为就是终止进程。

这个机制确保了管道命令的健壮性：当读取方不需要更多数据时，写入方会自动停止，避免浪费 CPU 资源。这体现了 Unix 设计哲学中的一个重要原则：*机制简单，后果可预测* 。
