读：Manager README 为什么不是好主意

Manager README 是技术管理圈一度流行的做法：管理者写一份文档说明自己的沟通偏好、工作风格、决策方式，让团队知道怎么跟你打交道。理论上能减少磨合成本，但 Camille Fournier（《The Manager's Path》作者）不买账。她在博客 Revisiting Manager READMEs 中重新审视了这个问题，态度明确：不推荐 README，更好的替代是轻量模板和沟通惯例。

Manager README 的问题在哪

Fournier 批评了两方面。

一是 自我中心 。README 的潜台词是"你们来适应我"。管理者当然需要让团队了解自己的关注点，但单向宣告偏好容易让人把精力花在管理上级上而不是做好工作本身。团队会按你的偏好包装信息而不是告诉你真正需要知道的事。

二是 懒惰 。高级管理者管辖面广，需要汇集各方信息来做判断，标准化信息传递是必要的。但 README 是最偷懒的做法：列几条偏好让团队去揣摩。真正有效的做法需要投入精力设计信息共享的标准，不只是列几条偏好。

更好的替代：轻量模板和沟通惯例

Fournier 的方案是 轻量模板和沟通惯例 （lightweight templates and ceremonies，即固定的沟通模板和例会机制）。她的实践是 Wins and Challenges （成果与挑战）：团队成员定期用统一格式汇报近期成果、当前挑战、需要的支持。格式固定，收集过程由团队自己决定，她只看结果。

信息格式统一了，横向比较和汇总都方便。关注点在把事说清楚，而不迎合上级偏好。模板要够轻才不会变成填表负担。找到合适的粒度需要几轮调整。她建议新到一个团队先学习 90 天，再引入流程，每次迭代观察 1-2 个季度。

不是所有事情都需要模板

同时，不要过度流程化。一位资深 CPO 给她的建议是，有些讨论需要开放空间，比如原型评审，开放式探索可能比固定议程更有效。她本人不给一对一设固定议程。有了议程就有必须聊完的压力，看到话题后也容易过早下结论。最好的讨论有时就是不预设地聊。

总结

几年前批评 README 时，Fournier 更多是指出问题。这次她给出了替代方案：轻量模板和沟通惯例，代替个人偏好说明书。README 让团队适应管理者个人，模板和沟通惯例让团队有一套共同的工作语言。前者是"我偏好什么"，后者是"我们怎么做才能更高效"。
