---
name: 故障回顾文章写作规范
description: 写故障排查回顾博文时必须遵守的叙事规范，避免逻辑跳跃和概念悬空
type: feedback
---

写故障排查回顾文章时：

1. **推理链不能有跳跃**：每一步结论必须有前文铺垫。不能突然出现一个结论说"推翻了之前的猜测"，如果前文从没提过这个猜测。不能突然用 `ss -ti dst 119.23.85.51`，如果没解释这个 IP 从哪来的。

2. **代词和指代要明确**：技术写作中"这""它"容易歧义。比如"这是个定时器"要写成"fd 14 是个定时器"，不说清楚是哪个 fd 读者会困惑。

3. **不能提前使用后面才得出的信息**：如果某个知识点（比如哪个 fd 是什么类型）是后面才查到的，前文就不能用它来做推理。叙事要按实际排查的时间线展开，读者跟着一步步走才不晕。

4. **排除法要解释理由**：说"排除了 XX"时必须解释为什么排除。比如 fd 是 inotify 所以 read 阻塞是正常的，但还要解释为什么这不能解释进程"卡住"的现象。如果有多个嫌疑对象（比如 fd 28 和 fd 15 都不触发），要解释为什么只怀疑其中一个而不怀疑另一个。

5. **消除无意义的步骤**：如果某个步骤（比如用 ps 确认进程状态）对排查没有实质贡献，就不要单独成节。可以用一句话带过。

**Why:** 故障回顾文章面向的是初学者，他们的上下文只有文章本身。作者脑中已有的结论和推理过程如果不写出来，读者就会遇到"怎么突然得出这个结论"的困惑，破坏阅读体验。

**How to apply:** 写完初稿后，从读者视角逐段检查：每个结论是否有前文支撑？每个"所以"前是否有足够的"因为"？每个专业术语首次出现时是否有解释？
